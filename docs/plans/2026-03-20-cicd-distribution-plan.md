## 1. Versioning Strategy

### Semantic Versioning

All releases follow [SemVer](https://semver.org/):

```
MAJOR.MINOR.PATCH
```

- **MAJOR** -- Breaking changes to user-facing behavior or data model
- **MINOR** -- New features (new detection engines, new quick actions, settings)
- **PATCH** -- Bug fixes, performance improvements, dependency updates

### Version Sources (Keep in Sync)

Three files must have matching versions:

| File | Field |
|------|-------|
| `package.json` | `"version"` |
| `src-tauri/tauri.conf.json` | `"version"` |
| `src-tauri/Cargo.toml` | `version` under `[package]` |

### Git Tags

- Format: `v0.1.0`, `v0.2.0`, `v1.0.0`
- Tags trigger the release workflows
- Tags are annotated, not lightweight:
  ```bash
  git tag -a v0.1.0 -m "v0.1.0"
  git push origin v0.1.0
  ```

### Changelog Generation

Use [git-cliff](https://git-cliff.org/) for automated changelog generation from conventional commits.

**Commit message convention:**

```
feat: add docker container scanning
fix: correct port detection on macOS Sequoia
perf: reduce scan interval CPU usage by 40%
docs: update README with Homebrew instructions
chore: bump sysinfo to 0.32
```

**git-cliff config (`cliff.toml` at repo root):**

```toml
[changelog]
header = """
# Changelog\n
All notable changes to localhost will be documented in this file.\n
"""
body = """
{% if version %}\
    ## [{{ version | trim_start_matches(pat="v") }}] - {{ timestamp | date(format="%Y-%m-%d") }}
{% else %}\
    ## [unreleased]
{% endif %}\
{% for group, commits in commits | group_by(attribute="group") %}
    ### {{ group | striptags | trim | upper_first }}
    {% for commit in commits %}
        - {% if commit.scope %}*({{ commit.scope }})* {% endif %}\
            {% if commit.breaking %}[**breaking**] {% endif %}\
            {{ commit.message | upper_first }}\
    {% endfor %}
{% endfor %}\n
"""
footer = """
---
Generated by [git-cliff](https://git-cliff.org/)
"""
trim = true

[git]
conventional_commits = true
filter_unconventional = true
split_commits = false
commit_parsers = [
    { message = "^feat", group = "Features" },
    { message = "^fix", group = "Bug Fixes" },
    { message = "^perf", group = "Performance" },
    { message = "^refactor", group = "Refactoring" },
    { message = "^doc", group = "Documentation" },
    { message = "^chore", group = "Miscellaneous" },
    { message = "^style", skip = true },
    { message = "^test", skip = true },
]
filter_commits = false
tag_pattern = "v[0-9].*"
```

**Usage:**

```bash
# Generate changelog for all releases
git-cliff -o CHANGELOG.md

# Generate changelog for a specific release
git-cliff --latest -o CHANGELOG.md

# Preview unreleased changes
git-cliff --unreleased
```

### Version Bump Script

`scripts/bump-version.sh` -- updates all three version files at once:

```bash
#!/usr/bin/env bash
set -euo pipefail

if [ -z "${1:-}" ]; then
  echo "Usage: ./scripts/bump-version.sh <version>"
  echo "Example: ./scripts/bump-version.sh 0.2.0"
  exit 1
fi

VERSION="$1"
ROOT="$(cd "$(dirname "$0")/.." && pwd)"

echo "Bumping version to $VERSION..."

# 1. package.json
cd "$ROOT"
pnpm pkg set version="$VERSION"

# 2. tauri.conf.json
cd "$ROOT"
tmp=$(mktemp)
jq --arg v "$VERSION" '.version = $v' src-tauri/tauri.conf.json > "$tmp" && mv "$tmp" src-tauri/tauri.conf.json

# 3. Cargo.toml (use sed -- version is near the top of the file)
cd "$ROOT"
sed -i '' "s/^version = \".*\"/version = \"$VERSION\"/" src-tauri/Cargo.toml

echo "Updated version to $VERSION in:"
echo "  - package.json"
echo "  - src-tauri/tauri.conf.json"
echo "  - src-tauri/Cargo.toml"

# 4. Generate changelog
if command -v git-cliff &> /dev/null; then
  git-cliff -o CHANGELOG.md
  echo "  - CHANGELOG.md"
fi

echo ""
echo "Next steps:"
echo "  git add -A"
echo "  git commit -m \"chore: bump version to $VERSION\""
echo "  git tag -a v$VERSION -m \"v$VERSION\""
echo "  git push origin main --tags"
```

## 2. Code Signing and Notarization

### Why This Matters

Without code signing, macOS Gatekeeper blocks the app. Users get the dreaded "app is damaged and can't be opened" dialog. For an unsigned app, users must manually allow it in System Settings > Privacy & Security, which is poor UX.

### Entitlements

localhost needs specific entitlements because it accesses process information and the network stack. Save as `src-tauri/entitlements.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <!-- Required: network access for Docker socket communication -->
    <key>com.apple.security.network.client</key>
    <true/>

    <!-- Required: network server to detect listening ports -->
    <key>com.apple.security.network.server</key>
    <true/>
</dict>
</plist>
```

**Note on process scanning:** The `sysinfo` crate reads from `/proc`-equivalent macOS APIs (sysctl, libproc). These do not require special entitlements on macOS -- any app can read basic process information for the current user. Elevated (root) process info would need different approaches, but localhost only needs user-level process data which is sufficient for dev tool detection.

### Notarization -- Two Phases

**Phase 1: MVP (no signing, no notarization)**

Users install via DMG and manually bypass Gatekeeper:
- Right-click the app > Open (first launch only)
- Or: System Settings > Privacy & Security > Open Anyway

This is acceptable for an early developer tool with a small user base. The Homebrew cask will include a caveat about this.

**Phase 2: Signed and notarized (when user base grows)**

Requires an Apple Developer account ($99/year). When ready, add these GitHub Actions secrets:

| Secret | Description |
|--------|-------------|
| `APPLE_CERTIFICATE` | Base64-encoded .p12 Developer ID Application certificate |
| `APPLE_CERTIFICATE_PASSWORD` | Password for the .p12 file |
| `APPLE_SIGNING_IDENTITY` | e.g., `Developer ID Application: Efe Celik (TEAMID)` |
| `APPLE_ID` | Apple ID email for notarization |
| `APPLE_PASSWORD` | App-specific password (not account password) |
| `APPLE_TEAM_ID` | 10-character team identifier |

The `tauri-action` supports these natively via environment variables:

```yaml
env:
  APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
  APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
  APPLE_SIGNING_IDENTITY: ${{ secrets.APPLE_SIGNING_IDENTITY }}
  APPLE_ID: ${{ secrets.APPLE_ID }}
  APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
  APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
```

### Info.plist

`src-tauri/Info.plist` -- declares system-level metadata:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>LSUIElement</key>
    <true/>
    <key>NSHighResolutionCapable</key>
    <true/>
</dict>
</plist>
```

- `LSUIElement = true` -- hides the app from the Dock (menu bar app behavior).
- `NSHighResolutionCapable = true` -- enables Retina display rendering.

## 3. GitHub Actions Workflows

### 3.1 release-macos.yml

This is the primary release workflow. It builds universal macOS binaries for both Apple Silicon and Intel.

```yaml
# .github/workflows/release-macos.yml

name: Release macOS

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    permissions:
      contents: write
    strategy:
      fail-fast: false
      matrix:
        include:
          - target: aarch64-apple-darwin
            arch: arm64
          - target: x86_64-apple-darwin
            arch: intel

    runs-on: macos-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Install pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: Install Rust stable
        uses: dtolnay/rust-toolchain@stable
        with:
          targets: ${{ matrix.target }}

      - name: Rust cache
        uses: swatinem/rust-cache@v2
        with:
          workspaces: './src-tauri -> target'

      - name: Install dependencies
        run: pnpm install

      - name: Build the app
        uses: tauri-apps/tauri-action@v0
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          # Uncomment when Apple Developer signing is set up:
          # APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
          # APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
          # APPLE_SIGNING_IDENTITY: ${{ secrets.APPLE_SIGNING_IDENTITY }}
          # APPLE_ID: ${{ secrets.APPLE_ID }}
          # APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
          # APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
        with:
          tagName: ${{ github.ref_name }}
          releaseName: 'localhost ${{ github.ref_name }}'
          releaseBody: |
            ## What's New

            See the [CHANGELOG](https://github.com/efeecllk/localhost/blob/main/CHANGELOG.md) for details.

            ## Download

            ### macOS
            - **Apple Silicon (M1/M2/M3/M4)**: Download `localhost_*_aarch64.dmg`
            - **Intel**: Download `localhost_*_x64.dmg`

            Or install via Homebrew:
            ```bash
            brew tap efeecllk/localhost
            brew install --cask localhost
            ```

            ## First Launch (unsigned builds)

            Since the app is not yet notarized, macOS will block it on first launch:
            1. Right-click the app and select **Open**
            2. Click **Open** in the confirmation dialog
            3. The app will launch normally from now on

            Alternatively: System Settings > Privacy & Security > scroll down > click **Open Anyway**.
          releaseDraft: false
          prerelease: false
          args: --target ${{ matrix.target }}
```

### 3.3 update-homebrew.yml

Triggered automatically when a GitHub Release is published (by the macOS workflow above). Downloads the DMGs, computes SHA256 checksums, and pushes an updated cask formula to the Homebrew tap repo.

```yaml
# .github/workflows/update-homebrew.yml

name: Update Homebrew Tap

on:
  release:
    types: [published]

jobs:
  update-homebrew:
    runs-on: macos-latest
    # Only run if release contains a DMG (macOS release)
    if: contains(toJSON(github.event.release.assets.*.name), '.dmg')

    steps:
      - name: Get release info
        id: release
        env:
          TAG_NAME: ${{ github.event.release.tag_name }}
        run: |
          VERSION="${TAG_NAME#v}"
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: Download DMGs and calculate SHA256
        id: sha
        env:
          VERSION: ${{ steps.release.outputs.version }}
          REPO: ${{ github.repository }}
        run: |
          # ARM64 DMG
          ARM_URL="https://github.com/${REPO}/releases/download/v${VERSION}/localhost_${VERSION}_aarch64.dmg"
          echo "Downloading ARM64 from: $ARM_URL"
          curl -L -o arm64.dmg "$ARM_URL"
          ARM_SHA=$(shasum -a 256 arm64.dmg | cut -d' ' -f1)
          echo "arm_sha=$ARM_SHA" >> $GITHUB_OUTPUT
          echo "ARM64 SHA256: $ARM_SHA"

          # Intel DMG (x64)
          INTEL_URL="https://github.com/${REPO}/releases/download/v${VERSION}/localhost_${VERSION}_x64.dmg"
          echo "Downloading Intel from: $INTEL_URL"
          if curl -L --fail -o intel.dmg "$INTEL_URL"; then
            INTEL_SHA=$(shasum -a 256 intel.dmg | cut -d' ' -f1)
            echo "intel_sha=$INTEL_SHA" >> $GITHUB_OUTPUT
            echo "Intel SHA256: $INTEL_SHA"
          else
            echo "Intel DMG not found, will use ARM64 only"
            echo "intel_sha=" >> $GITHUB_OUTPUT
          fi

      - name: Clone Homebrew tap
        env:
          TAP_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
        run: |
          git clone "https://x-access-token:${TAP_TOKEN}@github.com/efeecllk/homebrew-localhost.git"

      - name: Update cask
        env:
          VERSION: ${{ steps.release.outputs.version }}
          ARM_SHA: ${{ steps.sha.outputs.arm_sha }}
          INTEL_SHA: ${{ steps.sha.outputs.intel_sha }}
        run: |
          cd homebrew-localhost

          if [ -n "$INTEL_SHA" ]; then
            # Multi-architecture cask
            cat > Casks/localhost.rb << 'CASK_EOF'
          cask "localhost" do
            arch arm: "aarch64", intel: "x64"

          CASK_EOF
            echo "  version \"$VERSION\"" >> Casks/localhost.rb
            echo "  sha256 arm:   \"$ARM_SHA\"," >> Casks/localhost.rb
            echo "         intel: \"$INTEL_SHA\"" >> Casks/localhost.rb
            cat >> Casks/localhost.rb << 'CASK_EOF'

            url "https://github.com/efeecllk/localhost/releases/download/v#{version}/localhost_#{version}_#{arch}.dmg"
            name "localhost"
            desc "macOS menu bar app that detects and displays running dev processes"
            homepage "https://github.com/efeecllk/localhost"

            depends_on macos: ">= :catalina"

            app "localhost.app"

            zap trash: [
              "~/Library/Application Support/com.localhost.dev",
              "~/Library/Caches/com.localhost.dev",
              "~/Library/Preferences/com.localhost.dev.plist",
              "~/Library/Saved Application State/com.localhost.dev.savedState",
            ]
          end
          CASK_EOF
          else
            # ARM64-only cask (fallback)
            cat > Casks/localhost.rb << 'CASK_EOF'
          cask "localhost" do
          CASK_EOF
            echo "  version \"$VERSION\"" >> Casks/localhost.rb
            echo "  sha256 \"$ARM_SHA\"" >> Casks/localhost.rb
            cat >> Casks/localhost.rb << 'CASK_EOF'

            url "https://github.com/efeecllk/localhost/releases/download/v#{version}/localhost_#{version}_aarch64.dmg"
            name "localhost"
            desc "macOS menu bar app that detects and displays running dev processes"
            homepage "https://github.com/efeecllk/localhost"

            depends_on macos: ">= :catalina"
            depends_on arch: :arm64

            app "localhost.app"

            zap trash: [
              "~/Library/Application Support/com.localhost.dev",
              "~/Library/Caches/com.localhost.dev",
              "~/Library/Preferences/com.localhost.dev.plist",
              "~/Library/Saved Application State/com.localhost.dev.savedState",
            ]
          end
          CASK_EOF
          fi

          echo "Updated cask file:"
          cat Casks/localhost.rb

      - name: Commit and push
        env:
          VERSION: ${{ steps.release.outputs.version }}
        run: |
          cd homebrew-localhost
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"

          git add -A
          git diff --cached --quiet || git commit -m "Update to v$VERSION"
          git push
```

### 4.3 GitHub Token for Tap Updates

The `update-homebrew.yml` workflow pushes to a different repo (`homebrew-localhost`), so it needs a Personal Access Token (PAT):

1. Go to GitHub > Settings > Developer settings > Personal access tokens > Fine-grained tokens
2. Create a token with:
   - **Repository access:** Only `efeecllk/homebrew-localhost`
   - **Permissions:** Contents (Read and write)
3. Add it as a secret named `HOMEBREW_TAP_TOKEN` in the `efeecllk/localhost` repo settings

## 6. Release Process -- Step by Step

### Automated Release Script

`scripts/release.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

if [ -z "${1:-}" ]; then
  echo "Usage: ./scripts/release.sh <version>"
  echo "Example: ./scripts/release.sh 0.2.0"
  exit 1
fi

VERSION="$1"
ROOT="$(cd "$(dirname "$0")/.." && pwd)"

echo "=== localhost release v$VERSION ==="
echo ""

# 1. Ensure clean working tree
if [ -n "$(git status --porcelain)" ]; then
  echo "ERROR: Working tree is not clean. Commit or stash changes first."
  exit 1
fi

# 2. Ensure we are on main
BRANCH=$(git branch --show-current)
if [ "$BRANCH" != "main" ]; then
  echo "ERROR: Must be on main branch (currently on $BRANCH)."
  exit 1
fi

# 3. Pull latest
echo "Pulling latest from origin/main..."
git pull origin main

# 4. Bump version
echo ""
echo "Bumping version to $VERSION..."
"$ROOT/scripts/bump-version.sh" "$VERSION"

# 5. Generate changelog
if command -v git-cliff &> /dev/null; then
  echo "Generating changelog..."
  git-cliff --tag "v$VERSION" -o CHANGELOG.md
fi

# 6. Commit
echo ""
echo "Committing version bump..."
git add -A
git commit -m "chore: release v$VERSION"

# 7. Tag
echo "Creating tag v$VERSION..."
git tag -a "v$VERSION" -m "v$VERSION"

# 8. Push
echo "Pushing to origin..."
git push origin main --tags

echo ""
echo "=== Release v$VERSION initiated ==="
echo ""
echo "What happens next:"
echo "  1. GitHub Actions builds macOS DMGs (ARM64 + Intel)"
echo "  2. DMGs are uploaded to GitHub Releases"
echo "  3. Homebrew tap is automatically updated"
echo "  4. Users can install via: brew upgrade --cask localhost"
```

## 7. Future Windows Distribution

### When to Enable

Enable Windows builds when:
1. The `sysinfo` process scanning code has been tested on Windows
2. Docker Desktop for Windows socket path is handled (`//./pipe/docker_engine`)
3. Windows-specific process detection edge cases are addressed (PowerShell, WSL)
4. The NSIS installer has been tested locally

### tauri.conf.json Windows Configuration

The `bundle.windows` section in `src-tauri/tauri.conf.json`:

```json
{
  "bundle": {
    "windows": {
      "webviewInstallMode": {
        "type": "embedBootstrapper"
      },
      "nsis": {
        "installMode": "currentUser",
        "installerIcon": "./icons/windows/icon.ico",
        "displayLanguageSelector": false,
        "startMenuFolder": "localhost",
        "minimumWebview2Version": "110.0.1531.0"
      }
    }
  }
}
```

**NSIS configuration explained:**

| Field | Value | Why |
|-------|-------|-----|
| `installMode` | `currentUser` | No admin privileges needed -- dev tools should not require elevation |
| `webviewInstallMode` | `embedBootstrapper` | Bundles WebView2 installer so users do not need to install it separately |
| `minimumWebview2Version` | `110.0.1531.0` | Ensures modern web API compatibility |

### Windows Build Artifacts

Tauri produces two installers for Windows:

| File | Format | Size (approx) | Use case |
|------|--------|---------------|----------|
| `localhost_0.1.0_x64-setup.exe` | NSIS | ~5 MB | Standard installer with UI |
| `localhost_0.1.0_x64_en-US.msi` | MSI | ~5 MB | Enterprise/GPO deployment |

### Windows Code Signing (future)

For trusted Windows distribution, sign the installer with an EV code signing certificate:

| Secret | Description |
|--------|-------------|
| `WINDOWS_CERTIFICATE` | Base64-encoded .pfx certificate |
| `WINDOWS_CERTIFICATE_PASSWORD` | Password for the .pfx file |

In the workflow:

```yaml
env:
  TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.WINDOWS_CERTIFICATE }}
  TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.WINDOWS_CERTIFICATE_PASSWORD }}
```

### Windows-Specific Code Changes Needed

Before enabling the Windows workflow, the following Rust code paths need attention:

1. **Docker socket path:** Change from `/var/run/docker.sock` to `//./pipe/docker_engine` on Windows
2. **Process cwd detection:** The `sysinfo` crate handles this cross-platform, but verify edge cases with WSL processes
3. **Terminal open command:** Use `cmd.exe /c start` or `powershell` instead of macOS `open -a Terminal`
4. **Editor open command:** `code.cmd` instead of `code` for VS Code on Windows
5. **Browser open command:** `start` instead of `open`
6. **Tray icon:** Windows uses `.ico` format -- ensure icon pipeline produces it
